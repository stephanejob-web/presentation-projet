① Linux — Ubuntu 24.04 LTS

Pourquoi Linux ?

J'ai choisi Ubuntu 24.04 LTS comme système d'exploitation de développement pour plusieurs raisons concrètes :

  • Stabilité : Ubuntu LTS (Long Term Support) bénéficie de 5 ans de support officiel. Pas de redémarrages forcés en plein travail, pas de mises à jour intempestives qui cassent l'environnement — c'est un OS pensé pour une utilisation continue et fiable.
  • Même OS que la production : Mon VPS de production tourne sous Ubuntu Server. Développer sur le même OS élimine les bugs "ça marche en local mais pas sur le serveur", qui surviennent fréquemment entre Windows et Linux.
  • Outils natifs : Docker, Git, Node.js, SSH — tous parfaitement optimisés sur Linux, sans couche de compatibilité (pas de WSL, pas d'émulateur). Les commandes que j'exécute en développement sont identiques à celles du VPS.
  • Performance : Accès direct aux ressources système sans hyperviseur. Le build Vite, Docker Compose et les tests s'exécutent plus rapidement que sur Windows.
  • Compétence valorisable : Maîtriser Linux (permissions, processus, services systemd, logs) est une compétence directement attendue en entreprise web.

③ NVM — Node Version Manager

Pourquoi NVM plutôt qu'une installation Node.js classique ?

  • Plusieurs projets = plusieurs versions : Certains projets nécessitent Node 18, d'autres Node 20 ou 24. NVM permet de basculer instantanément entre les versions sans réinstaller.
  • Version identique local/VPS : Je m'assure que la version Node.js de mon VPS et de mon poste de développement est exactement la même (v24.11.1). Zéro surprise au déploiement.
  • Pas de sudo pour npm : Avec NVM, les packages globaux s'installent dans le dossier utilisateur. Pas besoin de droits root.

Composants clés développés :

► Carte interactive géolocalisée (Leaflet + supercluster + Web Worker)

Affichage des églises sur OpenStreetMap. Le clustering des marqueurs est calculé dans un Web Worker pour ne pas bloquer le thread principal. Un clic sur un marqueur affiche la fiche de l'église.

► Formulaires validés (React Hook Form + Zod)

Validation déclarative côté client avec les mêmes schémas TypeScript que côté serveur (validation isomorphe). Cohérence garantie des règles de validation entre le front et le back.

► Authentification côté client (Axios + intercepteurs)

L'intercepteur de requête injecte automatiquement le JWT dans chaque appel API. L'intercepteur de réponse gère globalement les 401 (token expiré) et 403 (accès refusé).

► Routes protégées (React Router v7)

Routes protégées selon le rôle (SUPER_ADMIN, PASTOR, MEMBER). Un composant ProtectedRoute vérifie le token et le rôle avant d'afficher la page.

► TypeScript strict — zéro any implicite

Tout le code front-end est typé strictement. PhpStorm détecte les erreurs de types en temps réel, avant d'exécuter le code. Refactoring sécurisé : renommer une interface la renomme partout.

Conception de la base de données — Modèle Relationnel :

Avant de coder, j'ai conçu le modèle entité-relation de la base de données. LightChurch nécessite une modélisation complexe avec plusieurs entités :

  • admins : utilisateurs (id, email, password_hash, role ENUM, status ENUM, first_name, last_name, created_at)

Normalisation 3NF :

Les données sont décomposées pour éviter la redondance. Séparation churches / church_details : la table principale contient les colonnes fréquemment filtrées (status, location), la table détails stocke les informations moins utilisées dans les filtres. Même pattern pour events / event_details.

Index stratégiques :

  • Index SPATIAL sur la colonne location POINT → requêtes de proximité géographique (ST_Distance_Sphere)
  • Index FULLTEXT sur les noms et descriptions → recherche textuelle
