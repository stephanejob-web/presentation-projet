01Introduction

Un besoin réel,
une solution concrète.

Le Problème identifié

Les communautés chrétiennes manquent d'outils numériques dédiés

Les églises gèrent leur organisation avec des outils génériques inadaptés : groupes Facebook pour les annonces, tableurs Excel pour les membres, emails pour les événements. Aucune solution centralisée, sécurisée et pensée pour leurs besoins spécifiques n'existait.

La Solution

LightChurch — plateforme full-stack dédiée

Une application web complète qui centralise la gestion des membres, des événements, des annonces et permet aux pasteurs de se connecter entre eux. Développée de zéro, hébergée sur un VPS, déployée en production réelle sur lightchurch.fr.

Pourquoi ce projet ?

Issu d'une communauté chrétienne, j'ai observé directement ces problèmes de gestion quotidienne. LightChurch est né de l'envie de mettre mes compétences techniques au service d'un besoin humain et concret.

Objectifs techniques

Concevoir une architecture full-stack découplée, maîtriser le déploiement VPS avec Docker, implémenter une authentification sécurisée par rôles, et livrer un produit réellement utilisable en production.

Besoins utilisateurs

Gestion des profils membres, publication d'événements, localisation d'églises sur carte interactive, réseau de pasteurs, validation des comptes via document SIRENE, panel d'administration complet.

Acteurs et rôles identifiés

SUPER_ADMIN

Accès total. Valide les comptes pasteurs (vérification document SIRENE), gère tous les utilisateurs, toutes les églises, tous les événements.

PASTOR

Gère son profil et son église. Crée des événements. Visible dans le réseau pastoral. Compte validé par le SUPER_ADMIN après vérification SIRENE.

Public (non connecté)

Accès à la carte publique des églises géolocalisées. Peut s'inscrire en fournissant son document SIRENE.

Avant de coder

Définir qui fait quoi

Écrire les User Stories m'a forcé à identifier les 3 rôles (Public, Pastor, Admin) et leurs besoins distincts avant de toucher au code. Ça a directement influencé le schéma BDD et le système de rôles JWT.

Priorisation

Savoir quoi développer en premier

Les User Stories permettent de prioriser : d'abord l'authentification (bloque tout), ensuite la gestion église (cœur produit), puis les événements. Sans ça, on code dans le désordre.

Critères d'acceptation

Savoir quand c'est terminé

Sans critères d'acceptation, une feature n'est jamais vraiment finie. Avec eux, je sais exactement quand un Use Case est complet — et je peux tester chaque scénario d'erreur systématiquement.

02dUX/UI & Design

Concevoir l'interface
à l'ère de l'IA.

Projet solo. Pas d'équipe design. Mais une démarche réfléchie — et des outils qui changent les règles.

Concevoir seul, sans équipe design

LightChurch est un projet réalisé entièrement en solo. Dans un contexte d'équipe, la démarche UX/UI classique passe par Figma : wireframes, maquettes haute fidélité, composants partagés, feedback itératif entre designers et développeurs.

Seul, cette phase aurait représenté plusieurs semaines de travail supplémentaires. J'ai choisi une approche différente — plus directe, et plus adaptée à la réalité d'un développeur solo en 2025.

WORKFLOW SOLO ADOPTÉ
01 Brief fonctionnel — lister les pages et les interactions nécessaires
02 Génération de prototypes HTML interactifs via IA (Claude / Anthropic)
03 Test UX direct dans le navigateur — ergonomie, animations, parcours utilisateur
04 Itération rapide — ajustements design en temps réel, intégration dans React

L'outil utilisé — Claude Design (Anthropic)

Claude d'Anthropic est le modèle d'IA qui a accompagné la conception des interfaces de LightChurch. En décrivant précisément les besoins fonctionnels — structure des pages, interactions attendues, hiérarchie visuelle — il devient possible de générer directement des composants HTML/CSS/React fonctionnels.

Ce n'est pas du "vibe coding" passif. C'est une collaboration dirigée : je spécifie les contraintes, je valide les sorties, je corrige les écarts avec les specs. L'IA exécute, le développeur décide.

CE QUE JE FOURNIS
  • ✦ Specs fonctionnelles précises
  • ✦ Contraintes techniques (React, TypeScript)
  • ✦ Retours et corrections itératives
  • ✦ Validation de chaque composant
CE QUE L'IA PRODUIT
  • ✦ Composants HTML/CSS/React
  • ✦ Animations et transitions
  • ✦ Structures de pages complètes
  • ✦ Variantes de design à comparer

Pourquoi une maquette HTML est plus proche de la réalité
UX réelle, pas simulée

Tester les vraies interactions

Une maquette Figma simule un clic. Un prototype HTML réagit vraiment — hover states, transitions CSS, animations JavaScript, scroll behavior. L'ergonomie perçue en navigation réelle est différente d'un prototype cliquable.

Itération immédiate

Modifier, voir, ajuster

Changer une couleur, un espacement, un composant = modifier une ligne de CSS et actualiser le navigateur. Pas de handoff, pas d'export, pas de perte en translation entre le design et l'implémentation. Le feedback est instantané.

Design system vivant

Variables CSS = tokens design

Les variables CSS (--primary, --radius, --spacing) sont les design tokens du projet. Changer le thème = modifier 5 variables. Le design system est dans le code, pas dans Figma.

Vision — où va le design web à l'ère de l'IA

La distance design ↔ code se réduit

Pendant des années, la frontière entre designer et développeur était nette : le designer maquette dans Figma, le développeur implémente en code. Cette séparation créait des frictions — perte en translation, délais de handoff, divergences entre le design prévu et le résultat final.

L'IA générative commence à effacer cette frontière. Un développeur peut aujourd'hui décrire une interface, obtenir un prototype HTML fonctionnel, tester l'UX réelle, et itérer — sans changer d'outil ni attendre un designer.

ÉVOLUTION DU WORKFLOW
Brief → Figma → Handoff → Dev → Test UX
Brief → IA + Dev → Prototype HTML → Test UX immédiat

Une analyse équilibrée

FIGMA — TOUJOURS ESSENTIEL

Dans les équipes avec designers dédiés, Figma reste irremplaçable pour la collaboration, le design system partagé et la documentation des specs. L'IA ne remplace pas le designer — elle augmente le développeur.

IA — UNE TENDANCE DE FOND

Des outils comme Claude, v0 (Vercel) ou Figma Make intègrent déjà la génération d'UI par IA. La frontière entre maquette et code continue de se réduire. Les prochaines années verront probablement une redéfinition du rôle du designer web.

MA POSITION

Je sais utiliser les deux approches. Sur LightChurch, le contexte solo justifiait le prototype HTML. En équipe, je saurais travailler avec Figma et respecter un handoff design. L'outil s'adapte au contexte — pas l'inverse.

02bBase de données

14 tables, modèle
normalisé et optimisé.

Choix de MySQL 8

Base relationnelle InnoDB : intégrité référentielle (FOREIGN KEY), transactions ACID, contraintes CASCADE. MySQL 8 apporte le type POINT avec index SPATIAL pour la géolocalisation native — pas besoin de stocker lat/lng séparément.

Normalisation 3NF

Données décomposées pour éviter la redondance. Séparation churches / church_details : la table principale garde les données indexées, la table détails stocke les infos moins filtrées. Même pattern pour events.

Index stratégiques

Index SPATIAL sur les colonnes POINT pour les requêtes de proximité géographique. Index FULLTEXT sur les noms et descriptions pour la recherche texte. Index composites sur (status, role), (status, created_at) pour les filtres admin fréquents.

La table churches garde les colonnes indexées (location, nom). church_details stocke les infos rarement filtrées. CASCADE DELETE — détails supprimés avec l'église.

Horaires récurrents

Une église a plusieurs créneaux hebdomadaires. Chaque ligne = un type d'activité + jour + heure. FK sur activity_types. CASCADE DELETE.

Hiérarchie confessionnelle

Une union (ex: Fédération Protestante) regroupe plusieurs dénominations. Relation SET NULL — une dénomination peut exister sans union.

events → event_interests (1:N)

Intérêt et notifications push

Un utilisateur mobile (identifié par device_id) manifeste son intérêt pour un événement. Contrainte UNIQUE(event_id, device_id) — un intérêt par appareil max.

POINT + SPATIAL INDEX

Géolocalisation native MySQL

Colonnes location et event_location de type POINT. Colonnes latitude et longitude générées automatiquement depuis le POINT. Index SPATIAL pour requêtes de proximité.

Géolocalisation native — type POINT et colonnes générées

MySQL stocke les coordonnées dans un type POINT (binaire optimisé). Les colonnes latitude et longitude sont calculées automatiquement — pas de redondance, pas de désynchronisation possible.

03cReact — Pourquoi ce choix

Construire des interfaces
qui réagissent.

C'est quoi React ?

Une bibliothèque pour construire des interfaces

React est une bibliothèque JavaScript créée par Meta (Facebook) en 2013. L'idée centrale : diviser l'interface en petits composants indépendants et réutilisables. Chaque composant gère son propre état et s'occupe d'une seule responsabilité.

Analogie : React, c'est comme des briques LEGO. Chaque brique est un composant. On les assemble pour construire l'interface. Si une brique change, seule cette brique se met à jour — pas toute la page.

Avant React — le problème

Manipulation manuelle du DOM = code ingérable

Sans React, pour mettre à jour l'affichage on écrivait : document.getElementById('...'). Quand l'interface devient complexe (dashboard avec 20 éléments), synchroniser manuellement les données et l'affichage devient un cauchemar. React automatise ça.

Pourquoi React pour LightChurch

SPA + état complexe + réutilisabilité

LightChurch a un dashboard avec rôles multiples, une carte interactive, des formulaires dynamiques, un système d'authentification persistant. Tout ça en React : une seule page, navigation instantanée, composants partagés entre pastor et admin views.

Le Virtual DOM — comment React est si rapide

React compare l'ancien et le nouveau VDOM (diff), puis ne touche que ce qui a vraiment changé dans le vrai DOM du navigateur.

Concepts React que j'utilise dans LightChurch
COMPOSANTS & PROPS

Un composant = une fonction qui retourne du HTML (JSX). Les props sont les paramètres qu'on passe au composant, comme des arguments de fonction.

function ChurchCard({ name, city, photo }) {
// props reçus du composant parent
return (
<div className="card">
<img src={photo} />

HOOKS — useState & useEffect

Les Hooks sont des fonctions spéciales qui donnent des super-pouvoirs aux composants. useState mémorise une valeur. useEffect déclenche du code quand quelque chose change.

const [churches, setChurches] = useState([]);

Le Context API résout le problème du prop drilling — devoir passer des données à travers 5 composants intermédiaires juste pour atteindre un composant profond. Avec Context, n'importe quel composant peut lire les données directement.

03dTypeScript — Pourquoi ce choix

JavaScript avec
un filet de sécurité.

C'est quoi TypeScript ?

JavaScript + types = moins de bugs

TypeScript est un langage créé par Microsoft qui étend JavaScript. Il ajoute un système de types : on dit au compilateur ce que contient chaque variable. Si on fait une erreur (appeler une méthode qui n'existe pas, passer un mauvais type), TypeScript signale le bug avant même d'exécuter le code.

Analogie : TypeScript c'est comme un correcteur orthographique pour le code. JavaScript laisse tout passer. TypeScript dit "attention, ça va planter" avant le run.

Le problème sans TypeScript

Des bugs invisibles jusqu'à la production

En JavaScript pur, user.name ne plante que si user est undefined — et ça, on le découvre en prod. TypeScript force à gérer ce cas à la compilation. Zéro surprise en production.

Pourquoi pour LightChurch

Interfaces complexes + données API

LightChurch manipule des dizaines de types de données : Church, Event, User, Pastor… Sans TypeScript, impossible de savoir quelles propriétés existent sur chaque objet. Avec TypeScript, PhpStorm les propose automatiquement en autocomplétion.

JavaScript vs TypeScript — la différence concrète
❌ JAVASCRIPT — bugs invisibles
// On sait pas ce que contient 'church'
function displayChurch(church) {
console.log(church.nme);
// ↑ typo : 'nme' au lieu de 'name'
// → undefined en prod, aucun avertissement
}
// Appel avec mauvais type
displayChurch("Paris"); // string, pas un objet
// → crash runtime, vu qu'en prod
✅ TYPESCRIPT — erreurs à la compilation
interface Church {
id: number;
name: string;
city: string;
}
function displayChurch(church: Church) {
console.log(church.nme);
// ↑ Erreur TS : 'nme' n'existe pas sur Church
// → bug détecté AVANT d'exécuter
}

Ce qu'est Git

Un système de sauvegarde intelligent

Git enregistre l'historique complet de chaque modification du code. Si quelque chose se casse, on peut revenir à n'importe quel état précédent. C'est aussi ce qui permet à plusieurs développeurs de travailler sur le même projet sans s'écraser mutuellement.

Ce que j'ai compris

Un commit = une intention, pas une sauvegarde

J'ai compris qu'un commit ne doit pas regrouper dix changements différents. Chaque commit doit avoir une seule raison d'être — ajouter une fonctionnalité, corriger un bug. Un historique propre permet de lire l'évolution du projet comme un journal.

Pourquoi les branches

Isoler pour travailler sans risque

Une branche est une copie parallèle du code. Je développe une fonctionnalité sur sa propre branche, sans toucher au code stable. Si ça ne fonctionne pas, je supprime la branche. Si ça marche, je fusionne (merge) dans develop puis main.

branche develop (intégration)
branche feature (fonctionnalité isolée)
merge → push main = déploiement automatique

Git Flow — une convention de branches structurée
C'est quoi Git Flow ?

Un workflow normalisé pour organiser les branches

Git Flow est une convention inventée par Vincent Driessen. Elle définit quelles branches existent, à quoi elles servent et comment les fusionner. C'est un protocole partagé — si tout le monde suit les mêmes règles, personne ne se marche dessus.

J'utilise Git Flow sur LightChurch pour structurer mon workflow : nouvelles fonctionnalités isolées dans des branches feature/, stabilisation dans develop, et main représente toujours la production stable.

Règle 1

main = production

On ne commit jamais directement sur main. Seuls les merges de release/ ou hotfix/ y arrivent. Chaque merge sur main crée un tag de version et déclenche le déploiement CI/CD.

Règle 2

develop = intégration

develop reçoit toutes les features terminées. C'est le miroir du prochain état de main. Si develop est cassé, aucune release ne peut partir. C'est le filet avant la prod.

Règle 3

hotfix → main ET develop

Un hotfix corrige un bug en prod. Il naît de main (pour partir du code stable), et doit être mergé dans les deux : main (pour corriger prod) et develop (pour que le fix ne soit pas perdu dans la prochaine version).

🤔 C'est quoi une Promise ?

Une Promise = une promesse qu'une valeur arrivera plus tard. JavaScript n'attend pas — il continue à tourner pendant qu'une requête réseau s'exécute. La Promise dit "je te promets de te donner le résultat quand c'est prêt".

Analogie 🍕
Tu commandes une pizza. Le pizzaiolo te donne un ticket numéroté (= la Promise). Tu n'attends pas debout au comptoir — tu vas t'asseoir, fais autre chose. Quand ta pizza est prête, on t'appelle (= la Promise est résolue). Si le four explose → on t'appelle aussi mais pour dire "désolé" (= Promise rejetée → catch).
async / await — du sucre syntaxique
Avant (promesses nues)

setData(res.data)

})
Après (async/await)

await dit "attends le résultat avant de passer à la ligne suivante" — mais sans bloquer le navigateur. Le code semble synchrone, il ne l'est pas.

Backend · Sécurité upload

Magic bytes — vrai type, pas le nom du fichier

Renommer un .exe en .pdf trompe l'extension mais pas les magic bytes. La librairie file-type lit les premiers octets du fichier pour identifier son vrai format.

Pourquoi ce choix

Les documents SIRENE sont des pièces officielles. Un fichier malveillant déguisé en PDF pourrait être exécuté côté serveur. Vérification des magic bytes = sécurité réelle, pas cosmétique.

Ce que j'ai compris

Un PDF commence toujours par %PDF-, un JPEG par FF D8 FF. Ces octets ne mentent pas. C'est comme ça que les antivirus identifient les fichiers suspects.

Pourquoi j'ai fait

Pas du code copié-collé. Chaque choix a une raison, une compréhension, et une leçon apprise.

🌍
Proxy backend pour le géocodage geoService.ts appelle /api/geo → Express appelle data.gouv.fr
Pourquoi Le navigateur bloque tout appel direct à une API externe depuis React — c'est la politique CORS. Si j'appelle data.gouv.fr depuis le frontend, erreur systématique.
Compris CORS est une protection du navigateur, pas du serveur. Mon backend Express n'a pas cette restriction — il peut appeler n'importe quelle API externe librement. Le proxy résout le problème proprement.
Appris La différence fondamentale entre ce qui tourne dans le navigateur (soumis au CORS) et ce qui tourne sur un serveur (aucune restriction réseau). C'est un concept clé de l'architecture web.
📍
MySQL POINT + SPATIAL INDEX schema.sql — churches.location, events.event_location
Pourquoi Stocker lat/lng en deux colonnes DECIMAL sépare une donnée qui est une seule entité géographique. Impossible de créer un index géographique dessus.
Compris MySQL a un type POINT fait pour ça — une coordonnée GPS est une seule valeur, pas deux. Avec un SPATIAL INDEX, les requêtes "trouver toutes les églises dans ce rayon" sont optimisées nativement.
Appris Modéliser les données selon leur nature réelle, pas selon ce qui est le plus simple à écrire. Une coordonnée GPS n'est pas un nombre — c'est un point dans l'espace. Le schéma doit le refléter.
❤️
Optimistic Update sur les intérêts hooks/useEventInterestWeb.ts
Pourquoi Attendre la réponse API avant de mettre à jour le bouton crée un délai visible. Le bouton ne réagit pas immédiatement — l'utilisateur ne sait pas si son clic a été pris en compte.
Compris Les grandes applis (Instagram, Twitter) ne n'attendent pas l'API — elles mettent à jour l'UI immédiatement et font un rollback si ça échoue. 99% du temps ça marche → l'UX est parfaite. 1% d'échec → on revient silencieusement.
Appris L'UX et la technique ne s'opposent pas. Gérer l'état optimiste demande plus de code (sauvegarder l'état précédent, rollback), mais l'utilisateur a une expérience fluide. C'est ce trade-off qui distingue un bon produit.
🧵
Web Worker pour le clustering carte workers/clusterWorker.ts
Pourquoi JavaScript est mono-thread. Calculer les clusters de 500 marqueurs sur le thread principal = la carte gèle, les animations bloquent, le scroll ne répond plus pendant le calcul.
Compris Un Web Worker est un thread séparé du navigateur. Il fait le calcul lourd en parallèle, puis envoie le résultat au thread principal. L'interface reste 100% fluide pendant ce temps.
Appris Identifier ce qui est coûteux en CPU et l'isoler. Pas besoin d'optimisation prématurée — mais quand la carte gèle à chaque zoom, le problème est réel. Le Web Worker a résolu ça proprement.
🔑
JWT stateless — pas de session serveur middleware/authMiddleware.js
Pourquoi Les sessions serveur stockent l'état en mémoire. Chaque redéploiement Docker redémarre le conteneur — toutes les sessions sont perdues, tous les utilisateurs déconnectés.
Compris Un token JWT est auto-suffisant — il contient les infos (id, rôle, expiration) et est vérifié par signature. Le serveur n'a rien à stocker. Redémarrage Docker → zéro impact sur les utilisateurs connectés.
Appris L'architecture doit être compatible avec l'infrastructure. J'avais un CI/CD qui rebuild régulièrement — les sessions serveur étaient incompatibles avec ça. Choisir JWT n'est pas juste "moderne", c'est adapté à mon contexte réel.
🐳

Pourquoi docker compose down supprime tous les conteneurs, y compris MySQL. J'ai perdu des données de production en faisant ça. Une seule fois suffit pour ne jamais recommencer.
Compris docker compose stop backend frontend cible précisément les services à redéployer. MySQL continue de tourner, les données sont préservées. On peut rebuild et relancer sans aucun risque.
Appris Comprendre chaque commande avant de l'exécuter en production. "down" et "stop" semblent similaires — ils ne le sont pas du tout. Cette erreur m'a appris à lire la documentation et à tester sur un environnement staging avant la prod.

Solution ENUM MySQL + contrôle explicite de chaque statut dans la route login + redirection UI selon le statut retourné
05

Géocodage fiable pour toutes les adresses françaises

data.gouv.fr couvre bien la France mais peut être indisponible. Nominatim est international mais plus lent. Il fallait une stratégie automatique transparente pour l'utilisateur.

Solution Fallback chain TypeScript avec timeout 10s, retry automatique sur Nominatim, statut de l'API utilisée exposé
06

CORS — les requêtes frontend bloquées par le navigateur

En développement, le frontend tourne sur localhost:5173 et le backend sur localhost:3000. Le navigateur bloquait toutes les requêtes API avec une erreur CORS policy: No 'Access-Control-Allow-Origin'. En production via Nginx/Traefik, tout passait par le même domaine — mais en développement, c'était bloqué.

Solution Proxy Vite en dev (toutes les requêtes /api redirigées vers le backend) + middleware cors() Express configuré avec les origines autorisées
Ce que j'ai appris : comprendre la Same-Origin Policy du navigateur — c'est une protection de sécurité, pas un bug. Il faut la contourner proprement, pas la désactiver.
07

Healthcheck MySQL — le backend démarrait avant la base de données

Au premier démarrage de Docker Compose, le backend Express tentait de se connecter à MySQL alors que le conteneur MySQL n'avait pas encore fini son initialisation (création des tables via schema.sql). Résultat : crash au démarrage avec ECONNREFUSED.

Solution Condition depends_on: mysql: condition: service_healthy dans docker-compose.yml + healthcheck mysqladmin ping avec retries=10
Ce que j'ai appris : "le conteneur démarre" ≠ "le service est prêt". Docker Compose a un système de healthcheck précisément pour ça.
08

TypeScript — passer d'un fichier JS existant à TS strict

J'ai voulu activer strict: true dans tsconfig.json en cours de développement. Résultat : 47 erreurs TypeScript en rouge partout — propriétés potentiellement undefined non gérées, props non typées, retours de fonction implicites. Décourageant au premier abord.

Solution Correction erreur par erreur, en comprenant chaque cas. Résultat : des interfaces solides pour toutes les entités API, plus aucun any implicite, et zéro bug de type en production
Ce que j'ai appris : les erreurs TypeScript ne sont pas des obstacles — ce sont des bugs découverts avant la production.
09

JWT — rester connecté après un rechargement de page

React recrée tout son état à chaque rechargement. Après un F5, l'utilisateur était déconnecté même si son token était encore valide. Le contexte React (AuthContext) était réinitialisé, le token en mémoire perdu.

Solution Persistance du token dans localStorage + useEffect au montage du AuthProvider qui relit le token et reconstruit la session
Ce que j'ai appris : la différence entre l'état React (volatile, en mémoire) et le stockage navigateur (persistant). Choisir où stocker les données selon leur durée de vie.

TypeScript — choix technique justifié
Pourquoi TypeScript et pas JavaScript ?

La sécurité des types à la compilation

JavaScript est dynamique — une erreur de type n'apparaît qu'au runtime, parfois en production. TypeScript détecte ces erreurs avant même d'exécuter le code. Sur un projet avec 13 pages, 9 modules API et des dizaines de composants, c'est essentiel.

Exemple concret : si l'API retourne user.role et que j'essaie d'accéder à user.roles (faute de frappe), TypeScript signale l'erreur immédiatement dans PhpStorm. En JavaScript, le bug silencieux passe en prod.

Ce que j'ai compris et appliqué

Typer les données, pas juste les variables

J'ai compris que TypeScript n'est pas juste "ajouter des types" — c'est modéliser les données de l'application. Chaque réponse API a son interface, chaque composant React ses props typées. Le compilateur devient un assistant qui valide la cohérence de tout le code.

J'ai appris la différence entre interface et type, les generics pour les services API réutilisables, et les types d'union pour les valeurs enum comme les rôles et statuts.

Docker — Conteneurisation

Un environnement identique partout

Pourquoi Docker ?

Reproductibilité garantie

Sans Docker, le classique "ça marche sur ma machine" est inévitable. Docker empaquète chaque service avec son environnement exact — version Node, MySQL, config Nginx. Le même conteneur tourne en dev et en production, sans surprise.

3 services orchestrés : MySQL 8 (données), Express 5 (API), Nginx+React (frontend). Traefik gère le reverse proxy, les certificats SSL et la redirection HTTP→HTTPS automatiquement.
Ce que j'ai compris

Isolation et orchestration

Chaque conteneur est un processus isolé avec son propre réseau, ses volumes, ses variables d'environnement. docker compose orchestre leur démarrage dans l'ordre correct avec les health checks.

Leçon concrète : docker compose down supprime les conteneurs. docker compose stop les arrête seulement. MySQL est préservé entre chaque déploiement — les données survivent.

docker-compose.yml — Orchestration des services

Chaque service est déclaré avec ses dépendances, ses volumes et ses variables d'environnement. depends_on avec condition: service_healthy garantit que MySQL est prêt avant que le backend démarre.

Chaque requête HTTP passe par Traefik (reverse proxy), qui route vers Nginx/React. Le frontend appelle l'API Express via le réseau Docker interne. Express interroge MySQL sur le même réseau isolé.
